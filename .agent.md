# NestJS Sample - Standardized Vulnerability Exposure Analysis

## Primary Evidence Files (Must Read First)

1. `docs/security/app-context.md`
2. `docs/security/architecture.md`
3. `docs/security/threat-model.md`

Do not infer exposure or criticality without checking these files.

## Core Principles

1. Repository documentation is the source of truth.
2. Do not assume internet-facing exposure.
3. Separate external exposure from internal/lateral movement risk.
4. Use technical severity and business context jointly.
5. If information is missing, state explicit assumptions.

## Risk Scoring Model

Score each factor 1 to 5:

- Exploitability
- Exposure
- Criticality

Formula:

`Risk = (Exploitability * 0.4) + (Exposure * 0.3) + (Criticality * 0.3)`

## OCIO Classification

- CRITICAL: active exploitation, or high-impact RCE/high exposure -> patch in 72 hours
- HIGH: serious vulnerability with restricted exposure/mitigations -> patch in 14 days
- MEDIUM: authenticated or limited impact scenarios -> patch in 30 to 90 days
- LOW: minimal impact or strong mitigating controls

Missed timelines require documented risk acceptance.

## Required Output Format (Always)

1. Application Overview
2. Vulnerability Summary
3. Scoring (Exploitability / Exposure / Criticality)
4. Risk Calculation
5. OCIO Classification (with patch timeline)
6. Key Interpretation (why this makes sense)
7. Required Actions
8. Justification Statement (audit-ready)

## Expected Workflow

1. Read the three security documentation files.
2. Determine actual deployment exposure and business impact.
3. Analyze advisory/CVE technical behavior.
4. Apply risk scoring model.
5. Map to OCIO class and timeline.
6. Produce concise one-page analysis using required output format.
7. Add practical patch steps when feasible.

## Patch Execution Guidance

1. Identify fixed version(s) and compatibility constraints.
2. Update `package.json` and refresh lockfile with `npm install`.
3. Validate with:
   - `npm test`
   - `npm run test:e2e`
   - `npm run lint`
   - `npm run build`
4. Confirm vulnerability closure (Dependabot alert state and/or `npm audit`).
5. Document CVE, scoring rationale, timeline, and any risk acceptance requirement.

## Constraints

1. Never manually edit `package-lock.json`.
2. Preserve Node.js 24 compatibility.
3. Preserve semver compatibility unless change is explicitly planned.
4. Avoid emergency merges without successful tests.
